## 威胁

### T1 — Access token 重放

**场景**：攻击者捕获有效 JWT 然后重用

**缓解**：
- JWT 短 TTL（默认 5 分钟）
- `jti` claim 等于 intent_id。中间件在短期缓存中记录已消费的 `jti`，TTL 窗口内拒绝重复
- 高价值 endpoint 提供方可设 `quotaCalls: 1`，让 token 单次使用

### T2 — 伪造 402 响应（恶意 endpoint）

**场景**：恶意 endpoint 返回一个收款地址自己控制的 402，然后即使没付款也返回 200

**v0.1 状态**：未完全防御。缓解：
- Agent 强制 `maxPricePerCall` 预算，单次损失有限
- 结算服务跟踪每个 endpoint 注册的 intent；如果 402 中的收款方与该 `endpoint_id` 注册的收款方不匹配，结算拒绝创建 intent

**v0.2 状态**：402 响应将用 endpoint 注册的 key 签名，agent SDK 验证签名

### T3 — 结算服务被攻破

**场景**：攻击者偷到结算服务的签名 key，伪造 access token

**缓解**：
- Key 存在 Vercel 加密环境变量；检测到泄露立刻轮换
- Key 轮换通过 JWT 的 `kid` header 和 JWKS 多 key 支持
- 纵深防御：高价值 endpoint（MVP 后）将要求链上证明，不只是 JWT

### T4 — Agent 钱包被攻破

**场景**：Agent 私钥泄露，USDC 余额被抽

**缓解**（agent 侧，非协议）：
- 建议 agent 只持有运营预算，从冷钱包补充
- `@tollgate/agent` 支持 `maxTotalSpend` 会话级
- 用 Privy embedded wallet 的 agent 受益于 Privy 的 MPC key 管理

### T5 — Webhook 伪造

**场景**：攻击者发假的 `webhook/helius` 事件，标记 intent 已付款而无实际链上支付

**缓解**：
- Helius webhook 带签名 HMAC。我们用季度轮换的共享密钥验证 HMAC
- Webhook handler 还会从 Solana RPC 重新拉取交易独立验证金额和收款方
- IP 白名单（Helius 公布出口 IP）

### T6 — 双花 / 竞态

**场景**：Agent 同时把同一笔 Solana tx 用于两个 intent，希望用一笔付款换两个 token

**缓解**：
- 每个 intent 通过 Solana Pay 嵌入唯一 `reference` 公钥。同一笔 tx 不能引用两个公钥
- 结算服务在 DB 上对 `(tx_signature, intent_id)` 加唯一约束
- 应用层竞态（同一 challenge 并行创建两个 intent）通过 `intents.challenge` 的 `UNIQUE` 索引防止

### T7 — Intent 枚举/耗尽

**场景**：攻击者创建数百万 intent 耗尽数据库或寻找 `challenge` 碰撞

**缓解**：
- `POST /v1/intent` 按 IP 和 agent_pubkey 限流（默认 10/分钟）
- `challenge` 是 UUIDv4（122 位熵）；碰撞统计上不可能
- `expires_at + 1 小时`后的 pending intent 被 GC

### T8 — 供应链（npm 包被攻破）

**场景**：有人攻破 `@tollgate/middleware` 或 `@tollgate/agent`，发布恶意版本

**缓解**：
- Provenance：GitHub Actions 构建用 `npm publish --provenance` 发布
- npm 账号 2FA
- GitHub 的不可变 release tag

### T9 — 支付提交被前置交易

**场景**：mempool 观察者前置 agent 的支付提取价差，或重放它

**缓解**：
- Solana 没有 EVM 意义上的公开 mempool；交易直接走到 leader
- 支付是固定金额 USDC 转账到固定收款方——没有滑点可利用
- `reference` 把支付绑到特定 intent；重放同一 tx 不能给另一个 intent 解锁

### T10 — Dashboard 用户钓鱼

**场景**：dashboard 用户被骗换了收款 Solana 钱包，所有收入被劫

**缓解**：
- Dashboard 对敏感修改（收款地址、splits）要求重新认证
- 任何收款方变更触发邮件 + 应用内通知
- 新收款人的 24 小时冷却期才能收钱（MVP 后）

## 加密选择

| 用途 | 算法 | 说明 |
|---|---|---|
| JWT 签名 | EdDSA (Ed25519) | 匹配 Solana 原生曲线；快速验签 |
| Webhook HMAC | HMAC-SHA256 | Helius 标准 |
| 随机 token | crypto.randomUUID() (v4) | 122 位熵 |
| 去重 hash | SHA-256 | 标准 |
