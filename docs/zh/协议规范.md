# Tollgate-402 协议规范

**版本**：0.1.0
**状态**：草案
**最后更新**：2026-04-29

> 本文档为 [PROTOCOL_SPEC.md](../PROTOCOL_SPEC.md) 的中文版。技术术语（HTTP 头、JWT 字段名、JSON 字段）保持原文。

---

## 1. 概述

Tollgate-402 是一个 HTTP 原生支付协议，使用状态码 `402 Payment Required` 表示付费资源需要在 Solana 上完成稳定币支付才能访问。它面向机器对机器（即"agent 对 API"）的商业场景设计。

## 2. 术语

- **Endpoint**：需要付费的 HTTP 资源（如 `https://api.example.com/v1/search`）
- **Provider**：endpoint 的运营方
- **Caller**：发起 HTTP 请求的实体（通常是 AI 智能体）
- **Settlement service（结算服务）**：观察到有效支付后签发 access token 的可信中介
- **Access token**：短生命周期的 JWT，由结算服务签名，授权 caller 访问指定 endpoint
- **Intent**：服务端记录的待处理支付（价格、收款方、过期时间）

## 3. 流程

### 3.1 未认证请求

```
Caller → Endpoint:
  GET /v1/search?q=Solana HTTP/1.1
  Host: wikipedia-pro.example.com
```

### 3.2 402 响应

如果 endpoint 收费且没有有效的 `Authorization: Bearer <jwt>` 头：

```
Endpoint → Caller:
  HTTP/1.1 402 Payment Required
  Content-Type: application/json
  Cache-Control: no-store

  {
    "tollgate": {
      "version": "0.1",
      "settlement": "https://tollgate.dev/api/settle",
      "endpoint_id": "wiki-search-v1",
      "price": "0.0005",
      "currency": "USDC",
      "network": "solana-mainnet",
      "challenge": "01f1c8c1-1a2b-4c3d-8e9f-0a1b2c3d4e5f",
      "expires_at": "2026-04-29T12:30:00Z"
    },
    "error": "Payment required",
    "doc": "https://tollgate.dev/docs"
  }
```

`challenge` 是服务端生成的 nonce，作为 Solana Pay 的 `reference` 字段、并作为结算服务内的唯一键。

### 3.3 Caller 创建 payment intent

```
Caller → Settlement:
  POST /v1/intent HTTP/1.1
  Content-Type: application/json

  {
    "endpoint_id": "wiki-search-v1",
    "challenge": "01f1c8c1-...",
    "agent_pubkey": "BDqnQu..."
  }

Settlement → Caller:
  HTTP/1.1 200 OK
  Content-Type: application/json

  {
    "intent_id": "int_a1b2c3...",
    "pay_url": "solana:BDqnQu...?amount=0.0005&spl-token=EPjFW...&reference=01f1c8c1-...&label=Tollgate&memo=wiki-search-v1",
    "expires_at": "2026-04-29T12:30:00Z"
  }
```

### 3.4 Caller 提交支付

Caller 构建一笔 Solana 交易：转账 `0.0005 USDC` 到 `BDqnQu...`（或多个收款人按 splits 分），把 `reference` 公钥作为只读账户嵌入，用自己的 keypair 签名并提交。

### 3.5 结算服务观察到支付

Helius webhook 把交易推给结算服务。结算服务验证：

- 收款地址和金额与 intent 一致
- 交易已确认（建议 `finalized` commitment）
- `reference` 匹配某个活跃 intent
- 付款钱包与 `agent_pubkey` 一致（如果指定）

如果有效，结算服务把 intent 标记为 `paid`，准备签发 access token。

### 3.6 Caller 用 intent 兑换 token

```
Caller → Settlement:
  POST /v1/confirm HTTP/1.1
  Content-Type: application/json

  {
    "intent_id": "int_a1b2c3..."
  }

Settlement → Caller（如果已支付）:
  HTTP/1.1 200 OK
  Content-Type: application/json

  {
    "access_token": "eyJhbGciOi...",  // JWT
    "token_type": "Bearer",
    "expires_in": 300,
    "tx_signature": "5Hf..."
  }
```

如果支付尚未确认：

```
HTTP/1.1 202 Accepted
{ "status": "pending", "retry_after": 1.0 }
```

### 3.7 Caller 用 token 重试

```
Caller → Endpoint:
  GET /v1/search?q=Solana HTTP/1.1
  Authorization: Bearer eyJhbGciOi...

Endpoint → Caller:
  HTTP/1.1 200 OK
  ...
```

Endpoint 中间件用缓存的 JWKS 验证 JWT 签名，检查 `aud`（受众）字段匹配自己的 `endpoint_id`，检查 token 未过期。

## 4. Access Token 格式

Tollgate access token 是 JWT，用 EdDSA（Ed25519）签名。结算服务在 `GET /v1/jwks` 发布公钥（标准 [RFC 7517](https://datatracker.ietf.org/doc/html/rfc7517) 格式）。

**Header**：
```json
{ "alg": "EdDSA", "typ": "JWT", "kid": "tg-2026-04" }
```

**Payload**：
```json
{
  "iss": "https://tollgate.dev",
  "sub": "BDqnQu...",                  // agent 公钥
  "aud": "wiki-search-v1",             // endpoint_id
  "iat": 1714400000,
  "exp": 1714400300,                   // 默认 5 分钟 TTL
  "jti": "int_a1b2c3...",              // intent_id，用于防重放
  "tg": {
    "v": "0.1",
    "calls_remaining": 1,              // 配额制 token（v0.1 几乎总是 1）
    "tx": "5Hf..."                     // 确认支付的 Solana tx 签名
  }
}
```

## 5. Solana Pay 编码

Tollgate-402 复用 [Solana Pay 规范](https://docs.solanapay.com/spec)。3.3 步的支付 URL 严格遵循该规范：

```
solana:<recipient>?amount=<amount>&spl-token=<USDC_mint>&reference=<challenge>&label=<...>&memo=<endpoint_id>
```

这意味着任何 Solana Pay 兼容钱包（Phantom 移动端、Solflare 等）都可以充当付款客户端。`reference` 公钥至关重要：它让结算服务能区分支付属于哪个 intent。

多收款方（splits）支付时，agent 必须构造单笔交易包含多个 `Transfer` 指令，目标是各收款方，但都引用同一个 `reference` 公钥。结算服务验证金额总和与收款方集合匹配注册的 splits。

## 6. Headers（信息性）

Tollgate-aware 客户端**应**发送包含 `Tollgate-Agent/<version>` 的 `User-Agent`，例如 `Tollgate-Agent/0.1.0`。

Endpoint **可以**在响应中反映 Tollgate 版本兼容性：

```
Tollgate-Version: 0.1
```

## 7. 免费层和绕过

Endpoint 可以授予免费访问（如登录用户、低流量 IP）。实现可以在 402 逻辑之前自由处理；本规范只管付费路径。

## 8. 错误

Caller 行为错误时，结算服务返回标准 JSON 错误：

| HTTP | Code | 含义 |
|---|---|---|
| 400 | `invalid_intent` | intent body 格式错误 |
| 404 | `intent_not_found` | 未知的 intent_id |
| 410 | `intent_expired` | intent 已过 `expires_at` |
| 409 | `intent_consumed` | 该 intent 已签发过 token |
| 422 | `payment_mismatch` | 交易金额或收款方与 intent 不匹配 |
| 429 | `rate_limited` | 请求过多；请遵循 `Retry-After` |

## 9. 版本

协议版本出现在 402 body 的 `tollgate.version` 字段和 JWT 内部。主版本可能不兼容；次版本是增量的。

## 10. v0.2 待解决问题

- **签名 402 body**：今天恶意 endpoint 可以在 402 body 里虚报价格或收款方。v0.2 要求 endpoint 向结算服务注册公钥，并签名 402 body。Agent 可在付款前验证签名
- **流式订阅**：长生命周期 token，授予时间窗口内的访问权（不只是单次调用）
- **退款**：endpoint 收到确认支付后未交付时的标准退款流程
- **跨币种**：支持 Solana 上其他稳定币（USDT、EURC、PYUSD）
- **L2/L3**：明确支持 Confidential SPL Token（Arcium）做隐私支付
